1 Downscale to the cell grid320×200 image → 40×25 cells, 8×8 pixels each ▶

A C64 text screen is 40 columns × 25 rows. Every cell is an 8×8 block of pixels. So the very first move is to squash the source image down to exactly 40·8 × 25·8 = 320×200 pixels, letting the browser do smooth averaging. After this, each 8×8 cell holds the pixels the algorithm has to reproduce with a single character.

downscaled · grid overlaid

The grid lines mark cell boundaries. Everything from here on happens cell by cell.

2 Choose one background for the whole screenthe C64 shares a single background color across all cells ▶

Here's the constraint that shapes everything: in standard text mode a cell can only show two colors — its own foreground, plus a background that's global to the entire screen. So before touching individual cells, the converter picks that one shared background.

It quantizes every pixel to the 16-color VIC-II palette, finds each cell's most common color, then takes the color that "wins" the most cells. That's almost always the backdrop.

3 Per cell: pick the foreground colorthe most common color that isn't the background ▶

Now each cell gets exactly one foreground color. The converter histograms the cell's 16-color pixels and takes the most frequent one that isn't the background. Combined with the shared background, every cell is now reduced to just two colors.

each cell's foreground color

This is the color stored in "color RAM" on the real machine — one value per cell, 0–15.

4 Binarize: where should foreground go?each of the 64 pixels becomes a single on/off bit ▶

With two colors locked, each pixel asks one question: am I closer to foreground or to background? That yields a 64-bit pattern — a tiny black-and-white stencil of where the ink belongs in this cell. The converter calls this the want mask.

want-mask for every cell (white = foreground)

Click any cell below in step 5 to see its want-mask up close.

5 Match the closest ROM glyphfind the character whose shape disagrees least — Hamming distance ▶

Every character in the C64 ROM is also an 8×8 stencil. The converter compares the cell's want mask against all 256 glyphs and counts how many of the 64 positions disagree — that count is the Hamming distance. The glyph with the fewest disagreements wins. Draw it in the foreground color over the background, and the cell is done.

But raw Hamming has a blind spot: it treats every pixel independently, so a thin line can match a hollow "O" (or even vanish to blank) if the counts happen to work out. Real converters add a second metric to protect structure. Use the 2nd metric menu above to compare flat Hamming against edge fidelity (boundaries must line up), shape (ink must sit near ink, via a distance transform), orientation (a horizontal cell can't grab a vertical glyph), or auto — which classifies each cell (flat / line / region / texture) and applies whichever metric suits it.
